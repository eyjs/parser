7-Phase Pipeline Detail

Phase 1 Document Intelligence

전체 문서 사전 스캔 — 페이지별 최적 전략 결정 (get_pixmap 없이 30초 이내)

파일document_intelligence.py
입력fitz.Document (PyMuPDF)
분석text_density, garbled_ratio, table_count, image_count
출력DocumentStrategyReport — 페이지별 PageStrategy
전략 pymupdf_text (디지털) → apple_vision_ocr (저텍스트/깨짐) → vlm_full (최종 폴백) → skip (노이즈)
임계값 text_density < 0.05 → OCR, garbled_ratio > 0.15 → OCR, image > 3 → Surya

Phase 2 Profile Document

문서 복잡도 프로파일링 — simple / moderate / complex 분류

파일profile_document.py
기준테이블 비율, 이미지 비율, 폰트 다양성, 레이아웃 복잡도
출력DocumentProfile (complexity, recommended_parser)

Phase 3 Learn Noise Patterns

전체 문서에서 반복 헤더/푸터/페이지 번호 패턴 학습

파일noise_detector.py
방법전 페이지 텍스트 블록의 y좌표 + 텍스트 유사도 클러스터링
출력LearnedPatterns (header_patterns, footer_patterns)
효과매 페이지마다 "보험약관 | OO생명" 같은 반복 텍스트 자동 제거

Phase 4 Document Statistics

전체 문서의 평균 폰트 크기, 줄 간격 계산 → 헤딩 감지 기준선

용도avg_font_size: 헤딩 판별의 상대 크기 기준, avg_line_gap: 단락 분리 기준

Phase 5 Parallel Page Processing

페이지별 병렬 처리 — 분류, 추출, 구조화, 테이블 추출을 동시 수행

핵심 알고리즘

Document Intelligence

  • text_density = len(text) / page_area
  • garbled_ratio = 깨진 문자 비율 (CJK 범위 밖 + 제어문자)
  • density < 0.05 + 이미지 있음 → OCR 전략
  • garbled > 0.15 → OCR 전략
  • table 지표 > 0 → Surya 레이아웃 감지 활성

페이지 분류

  • COVER: page_idx ≤ 2, 블록 ≤ 10, 60%+ 중앙정렬, 큰 폰트
  • TOC: 목차 키워드 + 60% 짧은 줄 + 40% 숫자 종결
  • NOISE: 문자 < 10, 이미지 없음
  • SCANNED: 문자 < 50, 이미지 있음
  • MIXED: 이미지 면적 30-70%
  • DIGITAL: 나머지 (기본값)

테이블 추출 (3-전략)

  • Strategy 1: lines_strict — 테두리선 기반 (고정 tolerance)
  • Strategy 2: text — 텍스트 정렬 기반 (DPI-adaptive tolerance)
  • Strategy 3: Surya hint — TABLE 영역 bbox로 제한 추출
  • 필터: 면적 ≥80% → 레이아웃 테이블 제거
  • 필터: 열 ≥10 → 비정상 구조 제거
  • 필터: 평균 셀 < 3자 → 파편화 제거

DPI Adaptive Tolerance

  • 수식: tolerance = round(base_tolerance × page_dpi / base_dpi)
  • 범위: clamp [3, 25]
  • 예시: 72dpi → 5pt, 150dpi → 10pt, 300dpi → 21pt
  • lines_strict에는 미적용 (선 좌표가 정확)
  • text 전략에만 적용 (정렬 허용 오차)

OCR Multi-Pass

  • Pass 1: Apple Vision OCR (macOS 호스트)
  • Pass 2: 저신뢰 블록 → VLM 크롭 보정
  • 적응 재시도: garbled 비율 높으면 OCR → VLM 순차 폴백
  • Semaphore로 동시 OCR 워커 수 제한

헤딩 감지 (4-신호 가중합)

  • Surya label: 레이아웃 모델이 heading으로 분류
  • height ratio: font_size / avg_font_size > 임계값
  • regex: "제N조", "제N장" 등 법률 문서 패턴
  • position: 페이지 상단 + 짧은 텍스트
  • OCR 페이지: font_size=0 → Surya + regex에 의존

노이즈 패턴 학습

  • 전 페이지 상단/하단 블록 수집
  • 텍스트 유사도 + y좌표 클러스터링
  • N/2 이상 페이지에 반복 → 패턴 확정
  • 페이지 번호: 숫자만 / "- N -" 패턴 자동 감지

줄 병합 (형태소 분석)

  • Kiwi 한국어 형태소 분석기 사용
  • 줄 끝 품사가 조사/어미 → 다음 줄과 병합
  • 줄 끝 마침표/물음표 → 단락 분리
  • y좌표 간격 > avg_line_gap × 1.5 → 단락 분리

API Endpoints

MethodEndpoint설명응답
POST /api/parse PDF 업로드 + 파싱 시작 (단일/멀티파일) 202 — { task_id, task_ids[] }
SSE /api/parse/{id}/status 실시간 진행률 스트림 (재연결 시 catchup) progress / page / done / error 이벤트
GET /api/parse/{id}/result 파싱 완료 결과 조회 200 — { markdown, metadata, stats }
GET /api/parse/{id}/export 마크다운 / 원본 PDF ZIP 다운로드 200 — ZIP 파일
GET /api/parse/{id}/compare 버전 간 diff 비교 200 — { diff, versions[] }
GET /v1/health 헬스체크 (Docker healthcheck 사용) 200 — { status: "ok" }
GET /api/queue/status 작업 큐 상태 조회 200 — { pending, running, completed }

SSE 이벤트 형식

progress

  • message: "[3/7] Learning noise..."
  • pct: 0-100

page

  • page_num: 페이지 번호
  • markdown: 해당 페이지 마크다운

catchup

  • 재연결 시 누적 상태 스냅샷
  • 마크다운 제외 (REST로 별도 조회)

done / error

  • 완료 또는 에러 종료
  • message + pct

인증 체계

내부 요청 (Docker 네트워크)

  • CF-Connecting-IP 헤더 없음
  • 인증 없이 모든 API 접근 가능
  • healthcheck 포함

외부 요청 (Cloudflare Tunnel)

  • CF-Connecting-IP 헤더 존재
  • DOCFORGE_EXTERNAL_AUTH=true 시
  • X-Internal-Key 헤더 필요
